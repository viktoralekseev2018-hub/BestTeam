# Техническое задание
# WEB-AGENT — Кроссплатформенный фоновый HTTP-клиент

**Версия:** 1.1
**Дата:** 2026-03-22
**Статус:** Актуализация после ЛР №1 (выполнены Config/Logger/HttpClient, цикл опроса в разработке)

---

## 1. Цель разработки

Создание кроссплатформенного фонового клиента **WEB-AGENT**, выполняющего:

- получение инструкций от удалённого сервера по HTTP/HTTPS;
- выполнение локальных задач (запуск программ, выполнение команд);
- сбор результатов выполнения задач;
- передачу результатов на сервер;
- запись всех операций в файл журнала (лог).

---

## 2. Область применения

WEB-AGENT предназначен для автоматического удалённого управления задачами на клиентских машинах. Агент работает в фоновом режиме, не требует участия пользователя и запускается автоматически при старте системы.

## 4. Режим работы

- **Фоновый** (daemon / Windows Service);
- **Автоматический запуск** при старте ОС;
- **Периодический опрос** сервера заданий с интервалом, заданным в конфигурации.

### 4.1. Текущее состояние (март 2026)

| Компонент | Статус | Комментарий |
|---|---|---|
| Config | Реализован | Читает/валидирует `config.json`, сохраняет полученный `access_code` обратно в файл |
| Logger | Реализован | Поддерживает stdout + вращающийся файл, уровни логирования настраиваются |
| HttpClient | Реализован частично | Работают запросы `/wa_reg`, `/wa_task`, `/wa_result`, добавлены таймауты; retry/backoff в планах |
| Agent | Частично | `init()` и цикл опроса с логированием готовы, обработка задач (`handleTask`), пул потоков и отправка результатов в разработке |
| Тесты | Реализованы для ЛР №1 | Проверяются конфигурация, окружение и структура репозитория |

## 5. Протокол взаимодействия

- **Протокол транспортного уровня:** HTTP/HTTPS
- **Формат данных:** JSON (запрос/ответ), multipart/form-data (загрузка файлов)
- **Base URL сервера:** `https://xdev.arkcom.ru:9999`

### 5.1. API-эндпоинты

| Метод | Путь | Назначение |
|---|---|---|
| POST | `/app/webagent1/api/wa_reg/` | Регистрация агента, получение `access_code` |
| POST | `/app/webagent1/api/wa_task/` | Запрос задания от сервера |
| POST | `/app/webagent1/api/wa_result/` | Отправка результата задания + файлы |

## 6. Функциональные требования

### 6.1. Инициализация (FR-INIT)

| ID | Требование |
|---|---|
| FR-INIT-01 | Загрузка конфигурации из `config.json` |
| FR-INIT-02 | Валидация полей конфига (uid, server_url обязательны) |
| FR-INIT-03 | Проверка доступности сервера (ping/HEAD запрос) |
| FR-INIT-04 | Регистрация агента: отправить `uid` + `descr`, получить `access_code` |
| FR-INIT-05 | Повторная регистрация (code == -3) должна считаться успешной |

*Примечание: если `access_code` уже сохранён в `config.json`, агент может пропустить сетевую регистрацию и сразу переходить к опросу сервера.*

### 6.2. Получение задания (FR-TASK)

| ID | Требование |
|---|---|
| FR-TASK-01 | Периодический опрос сервера с интервалом `poll_interval_sec` |
| FR-TASK-02 | При каждом опросе передавать `uid` + `access_code` |
| FR-TASK-03 | При ответе `status=WAIT` — повторить через интервал |
| FR-TASK-04 | При ответе с заданием (`code==1`) — немедленно приступить к выполнению |
| FR-TASK-05 | Полученные поля задания: `task_code`, `options`, `session_id` |

### 6.3. Типы заданий (FR-TYPE)

| ID | task_code | Описание |
|---|---|---|
| FR-TYPE-01 | `EXEC` | Запуск исполняемого файла с параметрами |
| FR-TYPE-02 | `CMD`  | Выполнение системной команды через shell |
| FR-TYPE-03 | `FILE` | Передача файла с клиента на сервер |
| FR-TYPE-04 | `CONF` | Обновление конфигурации агента |

### 6.4. Выполнение задания (FR-EXEC)

| ID | Требование |
|---|---|
| FR-EXEC-01 | Запуск указанного процесса/команды |
| FR-EXEC-02 | Ожидание завершения с таймаутом |
| FR-EXEC-03 | Фиксация кода возврата процесса |
| FR-EXEC-04 | Сбор файлов результата из `result_directory` |
| FR-EXEC-05 | Параллельное выполнение до `max_parallel_tasks` заданий |

### 6.5. Передача результата (FR-RESULT)

| ID | Требование |
|---|---|
| FR-RESULT-01 | POST `/api/wa_result/` с полями: `uid`, `session_id`, `result_code`, `message` |
| FR-RESULT-02 | Прикрепление файлов в формате multipart/form-data |
| FR-RESULT-03 | Повтор при сетевой ошибке (3 попытки, exponential backoff) |

### 6.6. Логирование (FR-LOG)

| ID | Требование |
|---|---|
| FR-LOG-01 | Логировать: запуск, регистрацию, задания, выполнение, результаты, ошибки |
| FR-LOG-02 | Формат строки: `[YYYY-MM-DD HH:MM:SS] [LEVEL] описание` |
| FR-LOG-03 | Уровни: DEBUG, INFO, WARN, ERROR |
| FR-LOG-04 | Вывод в stdout и в файл одновременно |
| FR-LOG-05 | Потокобезопасная запись в лог |
| FR-LOG-06 | Ротация файла лога (5 МБ, 3 архива) |

### 6.7. Многопоточность (FR-THREAD)

| ID | Требование |
|---|---|
| FR-THREAD-01 | Параллельное выполнение нескольких заданий |
| FR-THREAD-02 | Лимит параллельных задач: `max_parallel_tasks` |
| FR-THREAD-03 | Потокобезопасное логирование |
| FR-THREAD-04 | Graceful shutdown по сигналу SIGINT/SIGTERM |

## 8. Конфигурация (config.json)

```json
{
  "uid": "agent-001",
  "descr": "web-agent",
  "server_url": "https://xdev.arkcom.ru:9999",
  "poll_interval_sec": 10,
  "task_directory": "./tasks",
  "result_directory": "./results",
  "log_file": "./agent.log",
  "log_level": "info",
  "access_code": "",
  "max_parallel_tasks": 4,
  "retry_count": 3,
  "retry_delay_sec": 5
}
```

Если поле `access_code` пустое, агент регистрируется на сервере, получает токен и сохраняет его обратно в `config.json` для последующих запусков.

### Описание полей

| Поле | Тип | Обязательное | По умолчанию | Описание |
|---|---|---|---|---|
| `uid` | string | Да | — | Уникальный идентификатор агента |
| `descr` | string | Нет | `"web-agent"` | Описание агента |
| `server_url` | string | Да | — | Базовый URL сервера |
| `poll_interval_sec` | int | Нет | 10 | Интервал опроса в секундах |
| `task_directory` | string | Нет | `"./tasks"` | Директория для файлов заданий |
| `result_directory` | string | Нет | `"./results"` | Директория для файлов результатов |
| `log_file` | string | Нет | `"./agent.log"` | Путь к файлу лога |
| `log_level` | string | Нет | `"info"` | Уровень логирования: debug/info/warn/error |
| `access_code` | string | Нет | `""` | Предварительно выданный токен (если указан, регистрацию можно пропустить) |
| `max_parallel_tasks` | int | Нет | 4 | Максимум параллельных заданий |
| `retry_count` | int | Нет | 3 | Количество повторных попыток |
| `retry_delay_sec` | int | Нет | 5 | Начальная задержка retry (сек) |

## 13. Глоссарий

| Термин | Определение |
|---|---|
| **UID** | Уникальный идентификатор агента (`uid` в config.json). Присваивается при настройке агента. Используется в каждом запросе к серверу. |
| **access_code** | Токен доступа, выдаваемый сервером при успешной регистрации. Передаётся в запросах `wa_task` и `wa_result`. После получения автоматически сохраняется в `config.json`. |
| **session_id** | UUID, идентифицирующий конкретную сессию выполнения задания. Выдаётся сервером вместе с заданием, возвращается вместе с результатом. |
| **task_code** | Код типа задания (EXEC, CMD, FILE, CONF), определяющий, что именно должен сделать агент. |
| **agent** | Экземпляр программы WEB-AGENT, запущенный на клиентской машине. Идентифицируется по UID. |
| **poll interval** | Интервал между последовательными опросами сервера в поисках нового задания. Задаётся полем `poll_interval_sec` в секундах. |
| **exponential backoff** | Стратегия повторных попыток: задержка между попытками удваивается с каждой неудачей (5с, 10с, 20с...). |
| **multipart/form-data** | Формат HTTP-запроса для передачи бинарных данных (файлов) вместе с текстовыми полями. Используется при отправке результатов. |
